   (`binary_heap_tests.rs`, `hashbrown_hashmap_tests.rs`).
3. **T3 — reconciled creation_count** (часть `98b5d83`). Семантический
   баг Phase L: `cap_inspect_at` дёргал `record_initial`
   (=creation+sample), и биндинг с wrap_from + N cap_inspect получал
   creation_count=N+1. Контракт пересмотрен: creation_count считает
   только конструкции, total_observed считает поток сэмплов,
   cap_inspect теперь зовёт только record_sample. 6 тестов в
   creation_count_tests.rs.
4. **T4 — `merge` subcommand** (часть `0254aef`). Заменяет fragile
   `jq` one-liner с cp1252 проблемами. Группирует по
   (file,line,col), суммирует counts, конкатенирует samples,
   reservoir-samples при превышении cap через детерминированный
   per-site LCG (seed = djb2(file) XOR line × 0x9e37…c15 XOR col ×
   0x6c62…142).
5. **T5 — `measure` subcommand** (часть `0254aef`). Был `not yet
   implemented`. Теперь one-command pipeline: wire → instrument →
   cargo bench --no-run + run binaries with CAPTRACK_DUMP_DIR + timeout
   → merge → uninstrument → unwire. `CleanupGuard` RAII с
   `disarm()` — uninstrument+unwire запустятся даже при panic.
   Bench-крейт автодетектится через `cargo metadata`, бинари ищутся
   в `target/{release,debug}/deps/<name>-<hex>{.exe}` с tie-break по
   mtime.
6. **T6 — `analyze` subcommand** (часть `0254aef`). Distribution-aware
   policy recommender. Каждый сайт классифицируется в 6 форм
   (UnimodalTight CV<0.10 → cap_from=max; UnimodalSpread → p95;
   Bimodal → median × 2.0; HeavyTail p99/p50>10 → p95 × 0.5;
   MostlyZero >50% нулей → p99 over non-zero; InsufficientData <30 →
   skip). Порядок классификации важен (Tight → Bimodal → HeavyTail
   иначе bimodal маскируется). `--write-policy` инжектит per-site
   policy в JSON прямо в M11-схему — apply уже её читает.
