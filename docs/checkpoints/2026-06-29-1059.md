## Decisions

- **Path B (Dylint plugin) was sufficient for the user's actual goal** —
  earlier in the session I mis-stated that the pipeline was incomplete.
  The plan §1.1-1.2 explicitly says vanilla production code + dhat
  measurement at allocator level, no instrument step needed. After the
  user clarified the desired workflow ("ванильный код, инструмент сам
  оборачивает и измеряет"), Path C was added to extend coverage to the
  captrack-backend route.
- **Direct `TrackedX::with_capacity_named` (not a macro) for instrument
  replacement.** Avoids macro resolution surprises; the absolute
  `::captrack::TrackedX` path works regardless of the user's `use` lines.
- **Hasher choice = CLI parameter, not heuristic.** No profile metric
  decides which hasher to use; the user picks via `--hasher fx|...`.
  Absolute paths emitted (`::fxhash::FxBuildHasher::default()`) so the
  plugin never has to add `use` lines.
- **Capacity policy = three orthogonal knobs (source, multiplier,
  rounding).** Locked by user choice over alternatives (single fixed
  rule vs full per-site policy in profile). Both global CLI defaults
  AND per-site overrides supported.
- **Backward-compat for profile JSON via `Option<...>` wrapping**, not
  magic zero values. Pre-M10 profiles parse cleanly with `None` for
  new fields, and M11's rule code does explicit `unwrap_or(fallback)`.

## Open questions

- **Flaky telemetry test** `tbtreeset_records_peak_and_count` —
  intermittent failure under parallel cargo test runs; likely a race in
  the global scc::HashMap registry; passes in isolation. Worth a
  follow-up investigation but not blocking any current work.
- **Live integration tests for `apply`/`instrument`** still `#[ignore]`d
  — they require nightly + cargo-dylint + the cdylib build (~5-10 min
  first run). Could be moved to a CI-only job rather than `#[ignore]`d.
- **dhat backend cannot produce per-site `mean`/`p99`** — currently
  fills `None`, M11 falls back to `peak`/`p95` respectively. Documented
  but a true mean for dhat would require parsing the heap snapshot
  differently.
- **Hasher rewrite for struct fields / fn return types** — not detected;
  user gets a compile error if the suggestion is accepted in such a
  context. Documented limitation; a future M would handle by walking
  more HIR parent shapes.
